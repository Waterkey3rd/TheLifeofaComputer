# GDD 补充模块：高性能事件调度与分发系统

## 1. 架构变更思路

- **废弃：** 玩家每次触发事件 -> 请求后端 -> 后端请求 OpenAI -> 等待返回 -> 解析 JSON -> 发送给前端。
- **重构为：** 开发者/AI 提前批量生成数百个事件并存入 JSON 文件 -> 后端启动时将 JSON 加载到内存缓存 -> 玩家触发事件 -> **调度引擎（Dispatcher）根据当前状态毫秒级检索并下发最佳事件**。

## 2. 事件存储目录与热更新设计

摒弃将所有事件塞进一个巨大 JSON 的做法，采用**分片管理**，方便维护和热更新。

### 2.1 存储结构

Plaintext

```
backend/data/events/
├── routine/                  # 日常事件库
│   ├── basic_campus.json     # 基础校园事件集 (上课、自习)
│   ├── dorm_life.json        # 宿舍生活事件集 (室友断电、泼水)
│   └── internet_surfing.json # 冲浪事件集 (下载木马、钓鱼邮件)
├── crisis/                   # 危机事件库
│   ├── hardware_fail.json    # 硬件崩溃
│   └── software_crash.json   # 系统蓝屏
└── plot/                     # 固定剧情库 (如第4天发生活费)
    └── fixed_story.json
```

### 2.2 热更新机制 (Hot-Reload)

- **原理：** 后端维护一个内存变量 `EVENT_POOL`。
- **实现：** 1. 提供一个隐藏的 API 接口 `/api/admin/reload_events`。
  2. 当你修改了 JSON 或往目录里塞了新的 JSON 文件后，直接调用此接口。
  3. 后端清空 `EVENT_POOL`，重新遍历并读取 `data/events/` 目录下的所有文件并加载到内存，**全程无需重启服务器，玩家毫无感知，实现无缝热更新。**

## 3. 核心机制：事件调度算法 (Dispatcher Algorithm)

为了实现“新奇感”和“逻辑严密”，我们需要像抽卡游戏一样设计**过滤（Filter）**和**加权抽取（Weighted Draw）**机制。

### 3.1 增强版 JSON Schema (新增调度字段)

在之前的结构上，加入 `prerequisites` (前置条件) 和 `weight` (权重)。

JSON

```
{
  "event_id": "evt_routine_water_01",
  "event_type": "routine",
  "tags": ["hardware", "dormitory"], // 标签，用于控制同类事件的出现频率
  "is_unique": true, // 是否为唯一事件（触发一次后本局不再触发）
  "weight": 100, // 抽取权重。基准为100，越高越容易被抽中
  
  "prerequisites": { 
    // 调度引擎的过滤规则，只有全部满足，该事件才会进入备选池
    "min_day": 2, // 第几天之后才会出现
    "max_day": 999,
    "required_health_hardware": "<= 80", // 硬件低于80才会触发
    "exclude_flags": ["has_bought_cooling_pad"] // 如果玩家买了散热架，就不触发此过热事件
  },
  
  "title": "手滑的咖啡",
  "description": "...",
  // ... (options 结构保持不变) ...
}
```

### 3.2 调度流水分步解析 (Code Logic)

当需要给玩家生成一个【日常事件】时，后端的 Dispatcher 将执行以下四步：

1. **第一步：池化 (Pooling)**

   从内存的 `EVENT_POOL` 中取出所有 `event_type == "routine"` 的事件。

2. **第二步：硬性过滤 (Hard Filter)**

   遍历事件库，淘汰不符合 `prerequisites` 的事件。

   - *比如：玩家的 `hardware_health` 是 100，那么要求 `<=80` 的事件就会被剔除。*
   - *剔除玩家存档历史记录（LocalStorage 传来的 `played_events` 列表）中且 `is_unique == true` 的事件，防止一生只应该发生一次的事件反复出现。*

3. **第三步：动态权重修正 (Dynamic Weighting - 产生新奇感的关键)**

   - **标签降权惩罚：** 如果玩家昨天刚触发过带有 `"dormitory"` (宿舍) 标签的事件，引擎将备选池中所有 `"dormitory"` 标签的事件 `weight` 临时乘以 0.2。**这样能极大概率保证玩家每天遇到的事件类型都在切换**，避免连续三天都是“室友泼水”。

4. **第四步：加权随机抽取 (Weighted Random)**

   根据最终计算出的 `weight` 集合，执行加权随机算法（如 Python 的 `random.choices(population, weights)`），抽中一个事件并返回给前端。

## 4. 防沉寂设计 (Fallback System)

因为采用了严格的条件过滤，可能会出现一种极端情况：**玩家的状态太奇特，导致备选池被过滤得一个事件都不剩（池子空了）。**

- **解决方案：** 设计一组 **"兜底事件" (Fallback Events)**。
- **特性：** 这类事件没有任何 `prerequisites`，且 `is_unique` 为 `false`，权重极低。
- **举例：** * “今天风和日丽，你的电脑安静地运行着，什么事情也没有发生。”（健康度无变化）
  - “你用电脑看了一下午网协的科普视频，感觉自己又变强了。”（赛博常识 +1）
- **逻辑：** 只有当正常抽卡池为空时，才会强制从兜底事件库中随机抽取，确保游戏流程绝对不会卡死。
