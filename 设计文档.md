# 游戏设计文档 (GDD)：网络开拓者协会-赛博校园生存指南

## 1. 游戏概述

- 一款结合科普教育与资源管理的模拟经营/生存文字游戏 。

- 玩家将扮演一名刚进入大学的新生，在充满各种“电脑刺客”（流氓软件、意外事故、黑心商家）的校园环境中努力保护自己的设备 。

- 

  **核心目的**：提升大学生的电脑软硬件保护意识，并硬核宣传本校“网络开拓者协会（网协）电脑诊所”的免费义诊服务，对比校内外无良维修店的坑人套路 。

- 

  **网协核心设定**：网协是由热心同学组成，用爱发电，提供力所能及的免费帮助，但它并不包治百病，也不是万能的官方避风港 。

------

## 2. 核心玩法与运行时机制 (Runtime Loop)

游戏底层采用**状态机（State Machine）**驱动，摒弃单调的纯文字点击，将每天划分为具有不同交互逻辑的四个阶段，增强模拟经营的策略深度 。

### 2.1 游戏模式

- 

  **标准模式（正常开学第一周）**：存活 7 天，每天施加递增的生存压力 。第 4 天固定触发“生活费到账”事件 。

- 

  **无尽模式（大学四年）**：无时间限制，事件完全由隐藏概率驱动，直至触发“宕机退学”或“破产”结局 。

### 2.2 每日运行循环 (The Daily Loop)

1. 

   **清晨（日常行为阶段）**：触发与生活习惯相关的概率事件（如：室友借电脑、网页弹窗） 。这些事件带来潜在风险，玩家在此阶段必须做出选项抉择 。

2. 

   **白天（主动操作阶段）**：游戏时间暂停，玩家获得 UI 的完全控制权 。可以打开【校园钱包】查看余额，进入【二手东外设商店】购物，或者打开【校园服务小程序】进行主动的电脑维护 。

3. 

   **傍晚（危机判定阶段）**：系统后台校验电脑健康度 。若任意一项健康度低于危险阈值（如 < 30%），则触发“危机事件”，此时玩家必须在各派系维修店或网协中做出抉择 。若健康良好，则安全度过 。

4. 

   **深夜（每日结算阶段）**：结合已装备的外设 Buff，扣除自然老化损耗（如积灰增加），结算金钱与精神状态，保存进度至 LocalStorage，天数 +1 。

## 3. 彻底解耦的实体设计：电脑与外设

为了方便后期横向拓展，游戏采用配置驱动（Config-Driven）架构，所有实体完全解耦。

### 3.1 电脑对象模型 (Computer Object)

开发者只需通过编写 JSON 或 YAML 配置文件即可新增机型，无需修改核心逻辑 。

- 

  `model_id`: 唯一标识符（例：lenovo_y9000p_2023） 。

- 

  `display_name`: UI显示的名称 。

- 

  `base_health_hardware` / `base_health_system`: 初始硬件与系统健康度上限 。

- 

  `cooling_material`: 散热材质枚举，如 silicone (硅脂) 或 liquid_metal (液金，自行拆机极度危险) 。

- 

  `is_memory_upgradeable`: 是否支持加装内存（影响商店购买与事件） 。

- 

  `is_apple_silicon`: 是否为 ARM 架构 Mac（影响软件兼容性恶搞事件） 。

- 

  `price_tier`: 维修价格基数乘区，越高端的电脑去普通店修越贵 。

### 3.2 外设商店系统 (Items)

在【白天】阶段可访问，花费财富值购买，存入“背包”后提供长期 Buff 或单次保命效果 。

- 

  **RGB散热支架 (¥80)**：每日结算时，硬件健康度的自然过热损耗减半 。

- 

  **杂牌移动硬盘 (¥150)**：消耗品，遭遇“数据丢失”高危危机时，自动抵消惩罚并销毁 。

- 

  **青轴机械键盘 (¥299)**：精神状态每日增加，但大幅增加室友抗议/拔电源事件概率 。

- 

  **赞助正版火绒 (¥0)**：免疫 80% 的“日常误触下载”引发的系统降级事件 。

- 

  **冰美式 (¥25)**：单次消耗品，立刻恢复大量精神状态 。

## 4. 属性与状态系统

属性分为公开数值与隐藏状态，构成相互制约的养成体系。

### 4.1 公开属性 (UI面板显示)

- 

  **电脑总健康度 (0-100%)**：由硬件、系统、存储、软件四项加权计算 。低于 50% 时触发 UI 红色警告及动画卡顿 。

- 

  **财富值 (¥)**：核心生存资源，Day 4 固定触发生活费到账 。

- 

  **赛博常识 (0-100)**：核心成长科技树，数值越高，日常事件中解锁的“高级自救选项”越多（如解锁进入PE系统选项） 。

- 

  **精神状态 (0-100)**：电脑卡顿或被无良商家坑钱时下降，修好时上升 。过低会导致操作失误率飙升 。

### 4.2 隐藏属性 (后台逻辑驱动)

- 

  `net_assoc_trust`：网协信任/好感度，积累可解锁隐藏的特价渠道 。

- 

  `scam_counter`：大冤种指数，记录去坑人店的次数 。

- 

  `machine_flags`：机器状态标签，如 `has_malware` (潜伏木马) 或 `dust_level` (积灰层数，逐日递增增加过热概率) 。

## 5. 派系与主动维护系统

危机爆发或玩家主动进入小程序维护时，可选择不同的派系机构 。

- 

  **网络开拓者协会（网协）**：由热心同学组成，免费义诊 。需排队预约（可能消耗半天时间），能解决力所能及的故障并大幅提升玩家“赛博常识”，但不是包治百病 。

- 

  **学服联想维修店**：纯正的刺客，立刻处理无需排队，但收费极高 。技术差（大概率引入新毛病），且有概率触发恶意推销杀毒软件 。

- 

  **公主坟华硕维修店**：隐藏的大神据点，距离远去一次消耗半天，常规价格高 。若 `net_assoc_trust` 达标，可触发隐藏剧情获得超低价主板级顶级修复 。

- 

  **校外普通维修店**：数值上的保底选项，价格和修复率均为中等 。

## 6. 高性能事件调度引擎 (Dispatcher)

为了解决在线 AI 生成接口的延迟问题并保证内容的高扩展性，游戏采用**离线预生成池 + 动态调度系统** 。

- 

  **数据载体与热更新**：事件被提前批量生成并按类别存放在 `data/events/` 目录的 JSON 文件中 。后端支持无缝热更新，读取新 JSON 无需重启服务器 。

- 

  **事件设计原则**：事件选项需包含概率分布 。内容必须遵循“无厘头校园日常 + 绝对严谨的电脑科学常识”的限制 。

- **核心调度算法**：

  1. 

     **硬性过滤 (Hard Filter)**：剔除不符合 `prerequisites`（前置属性条件）的事件，并剔除已发生过的不可重复事件 。

  2. 

     **动态权重修正 (Dynamic Weighting)**：根据玩家近期遭遇的历史标签（如昨天刚遇到“宿舍”事件），对备选池中相同标签的事件进行临时降权惩罚，确保极强的随机感和新奇感 。

  3. 

     **加权抽取**：执行加权随机算法抽中最终事件下发给前端 。

- 

  **防沉寂设计**：设置无条件且权重极低的“兜底事件”（Fallback Events，如“电脑安静运行，什么也没发生”），确保过滤池为空时游戏流程绝不卡死 。

- 

  **离线生成工具**：提供带参运行及交互式的 Python 终端脚本，通过调用兼容 OpenAI 格式的 API 批量生成事件，并支持终端内人工审查把控文本质量 。

## 7. UI与交互设计 (桌面 OS 隐喻)

UI 彻底抛弃传统网文框，采用现代简约的“桌面操作系统”设计语言，全平台自适应 。

- 

  **开场动画**：加载时黑屏，全屏展示网协 Logo 像主板自检一样闪烁，伴随“滴”声开机音效过渡到主界面 。

- **桌面布局**：

  - 

    **左侧边栏（性能监控）**：类似任务管理器，以动态进度条展示四大健康度、精神状态和当前机型 。

  - 

    **中央主视窗（事件区）**：所有事件以“系统提示框”、“下载框”或对话框的 UI 弹出 。

  - 

    **底部/侧边 Dock 栏**：包含校园钱包、外设商店、校园服务预约、我的背包 。

  - 

    **悬浮操作按钮 (FAB)**：独立放置在右下角的“结束今天”按钮，推动时间线 。

  - 

    **应用拓展**：内置“赛博百科”应用供玩家查阅防坑指南 。

- **动效反馈**：

  - 限时选项按钮内置颜色从蓝变红的倒退进度条 。
  - 扣除健康度时，屏幕触发轻微的 CSS transform 震动及红色滤镜 。
  - 属性提升时伴随绿色飘字与清脆音效 。

## 8. 技术架构与部署方案

采用纯粹的**前后端分离架构**，满足高拓展性与工业级规范。

- 

  **前端技术栈**：Vue 3 (Composition API) 或 React + Tailwind CSS 。使用 Pinia 或 Zustand 管理复杂状态，基于浏览器的 UUID 和 LocalStorage 存储独立存档，支持扫码即玩 。

- 

  **后端技术栈**：FastAPI (Python)，纯无状态（Stateless）API 设计，利用 Pydantic 严格约束 JSON 数据模型 。

- 

  **部署方案**：使用 Docker 容器化编排 (docker-compose)，内含 Nginx 容器（前端分发与反向代理）和 Uvicorn 容器（后端 API） 。

- 

  **Debug 与错误日志系统**：在主界面特定角落（如连续点击系统时间 5 次）可呼出“开发者终端” 。终端内可查看抓包数据，手动修改隐藏的 `machine_flags` 或一键加钱/扣血，极大方便 QA 测试 。
